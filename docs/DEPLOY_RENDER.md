# Dev Deployment — Render (Blueprint) + Atlas + Redis Cloud + Stripe TEST

Two Docker web services defined in `render.yaml`, deployed from a GitHub repo.
MongoDB Atlas + Redis Cloud are external managed stores; secrets are entered in
the Render dashboard (never committed).

```
            ┌── Render Blueprint: aviator ─────────────────────────┐
 testers ──▶│  aviator-crash-web (Next.js) ─HTTP/WS─▶ aviator-crash-api │──▶ Atlas (Mongo)
            │  *.onrender.com                          *.onrender.com   │──▶ Redis Cloud
            └──────────────────────────────────────────────────────┘──▶ Stripe (test)
```

## Step 0 — code on GitHub
The repo is pushed to GitHub (done via PAT). Render reads `render.yaml` from the root.

## Step 1 — create the Blueprint
1. Render Dashboard → **New → Blueprint**.
2. Connect the GitHub repo (authorize Render's GitHub app on it — one-time).
3. Render detects `render.yaml` and lists two services: `aviator-crash-api`, `aviator-crash-web`.
4. It prompts for the `sync: false` secrets — enter them now (see table).
5. **Apply** — both services build and deploy.

## Step 2 — secret env vars (backend service)
| Var | Where to get it |
|---|---|
| `MONGODB_URI` | Atlas → Connect → Drivers; append `/aviator`. Network Access → allow `0.0.0.0/0`. |
| `REDIS_URL` | Redis Cloud DB → public endpoint + password → `rediss://default:PASS@host:port` |
| `STRIPE_SECRET_KEY` | Stripe (TEST) → Developers → API keys → `sk_test_…` |
| `STRIPE_WEBHOOK_SECRET` | Set in Step 4 (leave blank or placeholder initially) |
| `ADMIN_EMAIL` / `ADMIN_USERNAME` / `ADMIN_PASSWORD` | your choice |

`JWT_SECRET` / `JWT_REFRESH_SECRET` are auto-generated by Render (`generateValue: true`).

## Step 3 — verify URLs
After first deploy, confirm the assigned URLs match what's in `render.yaml`:
- API:  `https://aviator-crash-api.onrender.com/health` → `{"status":"ok"}`
- Web:  `https://aviator-crash-web.onrender.com`

If Render appended a suffix (name was taken), update the three URL values in
`render.yaml` (`FRONTEND_URL`, `NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_SOCKET_URL`),
push, and redeploy. The frontend must be **rebuilt** for the public URLs to take
effect (they are baked at build time).

## Step 4 — Stripe webhook
1. Stripe (TEST) → Developers → Webhooks → Add endpoint:
   `https://aviator-crash-api.onrender.com/api/payments/webhook`
2. Event: `checkout.session.completed`.
3. Copy the signing secret (`whsec_…`) → backend `STRIPE_WEBHOOK_SECRET` → save (redeploys).

## Step 5 — seed admin & smoke test
- Admin auto-seeds on first boot if `ADMIN_*` are set (or run a one-off shell:
  Render → aviator-crash-api → Shell → `npm run seed:admin`).
- Smoke test: open the web URL, register, watch a round, place a bet, cash out,
  check the leaderboard, log in as admin → `/admin`.

## Free-tier caveats (dev only)
- **Cold starts / sleep:** free web services sleep after ~15 min without inbound HTTP.
  When the **backend** sleeps, the game loop pauses until the next request wakes it
  (~50s cold start). For continuous play during testing, upgrade `aviator-crash-api`
  to the **Starter** plan (`plan: starter` in `render.yaml`) or add an uptime pinger.
- **Single backend instance only** — it runs the authoritative game loop; do not scale
  the API service to multiple instances.
- Region is `singapore` (low latency for India-based testers); change if your testers
  are elsewhere.
- This is **dev/test**: Stripe TEST mode, play-money economy — not for real wagering.
